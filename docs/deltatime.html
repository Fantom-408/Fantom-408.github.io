DeltaTime Extension Documentation

Unlike custom extensions on other websites, this isn't limited by the extension sandbox, so it is a lot more powerful. All official TurboWarp extensions are reviewed for safety, including this.

To use this extension in TurboWarp, copy its URL from the gallery. Then go to the editor, open the extension chooser, then choose the "Custom Extension" option at the bottom, and enter the URL.

This extension is not compatible with Scratch.
Projects that use this extension can't be uploaded to the Scratch website. They can, however, be used in the packager.
This extension may not work in TurboWarp Desktop.
For compatibility, security, and offline support, each TurboWarp Desktop update contains an offline copy of this extension from its release date, so it may be outdated or missing. Use the latest update for best results.

This Extension Allows You To:

  • Make your project run at the same speed on all devices and in all its parts. While, this speed will be the one that you want!
  • Test the speed of your game under different circumstances!
  • Make using Turbo Mode more enjoyable!
  • Make a multiplayer game where players will not lag behind each other due to freezes!
  • And many more interesting things!

The extension helps you to use DeltaTime (the time in seconds it took to draw the last frame). So now we will rely not on our FPS (frames per second), but on the actual time, which is the same for all devices.

Main Blocks

The most important block for which the expansion was made. The symbol "Δ" is the Greek letter "Delta", and "t" is an abbreviation of "Time", so as you have already understood, this reporter returns cherished DeltaTime.

Returns your actual FPS. This value is not equal to the "FPS limit" of the project, which you can get using Runtime Options, most often it is slightly lower, depends on the device and sometimes likes to jump a lot :)

Simple Example

This example fully describes the main use of the extension:

Running this script, you will notice that the sprite on which it is placed constantly turns by 15 degrees. The catch is that the rotation always happens at the same speed, despite the lower block, which changes the "FPS limit", and therefore the speed of a normal project. Let's figure it out!

The fact is that now, instead of simple values of type 15, we will use a construction of type calculate(Δt) * 15. Using this is already enough to fulfill the goals set above, but let's go a little deeper.

  • The "Set FPS limit" block is needed only for clarity, it is not needed in the real project!
  • If you tested this example and saw that the sprite sometimes moves jerkily, this is normal, because thanks to the last block, your FPS may even be lower than 1. In real projects, this will not happen, so the animation will almost always be smooth, but by the way, despite these jerks, the sprite still does not lose speed! And that's exactly what we wanted to see ^-^

Calculation Blocks

You have already seen one of these blocks above, but the rest are new. So what are they all doing? The first 2 blocks, as the name implies, set and return the value of a certain "Calculation Standard". This value uses the 3 block. In fact, it just multiplies by it the number that you enter, yes, it's that simple :)

These blocks are only needed to shorten the code. As shown in the first example, you use not just DeltaTime, but calculated, that is, multiplied by a certain number, by default, 30. This number can be perceived as the "FPS limit", which, in terms of speed, the project will match perfectly.

  • "Calculation Standard" can be set to any positive value (even float number) and zero. If you enter a number below zero, it will automatically become 0.

Adaptation Example

If you run these 2 scripts separately, you will see that the sprite with them will turn at about the same speed. The second script may work slower, because it depends on the FPS, which we talked about earlier. It's the same with other similar scripts, the main thing is that the "FPS limit" and "Calculation Standard" are equal.

For example, if you want to integrate DeltaTime into your project with the "FPS limit" equal to 60, and not break anything, set the "Calculation Standard" value to 60. Also note that these 2 values do not have to be equal, BUT their difference only create inconvenience, so I do not recommend it.

  • If you have integrated DT into your project with 30 "FPS limit", you can most likely set "Calculation Standard" to 30 and "FPS limit" to 60, and make all smoother without breaking anything, but this is inaccurate because it depends on the project itself.

Wait Example

Now that we know the basics of using DeltaTime, it's worth clarifying one important detail. Surely you have already thought about the waiting blocks in Scratch, is it worth replacing them with something too? The answer is quite joyful: No!

The fact is that DeltaTime, as already mentioned, uses the actual time, exactly the same as in the waiting blocks. Look at the example above. These 2 scripts implement the timer through the standard blocks and through DT. If you test the scripts from the example above together, you will see that they give the same result (look at the values of the variables "Classic Wait" and "ΔT Wait"). So use the usual waiting from Scratch 3, because it's shorter, more familiar and loads your device less :)

  • The variable "i" in the first script is used to remember the value of DeltaTime.
  • In the first script, the value of the variable contains a fractional part, and in the second it does not. This is done for convenience, so as not to lengthen the script. If you want to get equal values, add set it to floor of "ΔT Wait". Note that these values will be slightly different in any case due to the features of Scratch, as well as lag behind the usual timer. That's why I recommend using More Timers instead of homemade timers.
